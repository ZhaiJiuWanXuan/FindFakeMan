# FindFakeMan 程序员实现规范

> 来源：`设计表.xlsx`
>
> 说明：本文档将设计表内容整理为程序实现约束，面向开发、联调和验收。若后续设计表更新，应同步维护本文档。

## 1. 总体架构原则

- 项目采用 **Manager + Services + 数据配置** 的组织方式。
- Core/Gameplay/Narrative/Samples 必须保持分层：Core 不依赖 Samples，Gameplay 不依赖 Samples。
- 跨系统调用优先通过 Manager 公共 API、Core 接口或 `Services.Get<T>() / Services.TryGet<T>()`。
- UI、调试脚本、Sample 场景只负责表现和演示，不承载核心业务规则。
- 运行时对象必须使用稳定 ID 关联配置、存档和场景状态。

## 2. Manager 职责规范

### GameManager

- 管理游戏全局状态：初始化、探索、调查、拖拽、暂停、结局等。
- 负责状态切换、胜利、失败、新游戏、继续游戏、快速存取档流程。
- 对外暴露 `CurrentState`、`PreviousState`、`OnStateChanged`。

### InteractionManager

- 管理所有可交互物的注册、注销、点击、工具拖拽和工具使用。
- 负责从交互输入切换到 Inspection / ToolDragging 等状态。
- 负责工具是否可用于目标、工具交互执行、ActionResult 回调。

### UIManager

- 只负责 UI 表现层 API：面板、检视、拖拽指示、SAN、证据、收容、提示、VN UI。
- 不负责交互判定、道具判定、分支判定。
- UIManager 不得依赖 Sample 具体类型；场景 UI 应通过 Core 接口注册。

### SanityManager

- 管理 SAN 值增减和归零判定。
- SAN 变化后通知 UI 更新和屏幕效果。
- SAN 归零时触发 GameManager 的 GameOver 流程。

### InventoryManager

- 管理物品和工具库存。
- 提供添加、移除、查询、装备工具等接口。
- 工具耐久变化后应触发装备 UI 刷新。

### FlagManager

- 维护全局 flag 字典，是剧情、分支、交互状态的唯一事实来源之一。
- 所有需要跨系统判断的布尔/数值状态应进入 FlagManager 或明确的数据状态对象。

### BranchManager

- 负责伪人/异常线索相关分支判定。
- 根据工具读数、概率、flag、配置决定真实/异常分支。
- 分支结果必须可追踪、可存档、可复现。

### EvidenceManager

- 统计已收集关键证据。
- 达到目标证据数量时触发对应事件或胜利检查。

### SaveManager

- 负责序列化/反序列化存档。
- 存档内容至少包括：SAN、库存、工具耐久、flag、当前场景、分支状态、已拾取/已破坏交互物列表。

### AudioManager

- 管理 BGM、环境音、UI 音效、角色语音和工具音效。
- 支持动态音量、淡入淡出和事件触发播放。

### SceneManager / GameLoopManager / CGMgr

- SceneManager 封装场景加载和加载进度回调。
- GameLoopManager 管理关卡内部核心循环。
- CGMgr 管理剧情图片和 CG，支持播放前后事件。

## 3. Item / Tool / Action 规范

### Item 基类

每个 Item 至少包含：

- `itemId`
- `displayName`
- `icon`
- `description`

建议公共行为：

- `OnPickup()`
- `OnDiscard()`
- `OnInspect()`
- `GetAvailableActions()`

实现要求：

- Item 不直接写 UI 或场景状态。
- Item 的交互行为应封装为 ItemAction。
- Item ID 必须稳定，用于配置、存档、日志和调试。

### ClueItem

ClueItem 用于证据、线索和伪人判断。

字段要求：

- `isAnomaly`：是否异常/伪人线索。
- `requiresContainment`：是否需要收容。

行为要求：

- 拾取异常线索时应通知 InventoryManager / EvidenceManager / FlagManager。
- 丢弃异常线索时可触发 SAN 惩罚。
- Inspect 时根据 `isAnomaly` 显示不同描述或提示。

### ToolItem

ToolItem 用于工具交互。

字段要求：

- `durability`
- `maxDurability`
- `toolType`
- `equippedSlotIndex`

行为要求：

- `OnEquip()` / `OnUnequip()` 更新装备状态和 UI。
- `OnUse()` 消耗耐久并执行工具效果。
- 工具耐久不足时应返回失败结果，不得继续执行业务效果。

### ItemAction

交互行为必须优先封装为 Action 类型，例如：

- `PickupAction`
- `DiscardAction`
- `EquipAction`
- `UseToolAction`
- `UseConsumableAction`
- `InspectAction`

实现要求：

- Action 内部可通过 Services 获取所需 Manager。
- Action 负责顺序执行业务：库存变化、SAN 变化、flag 设置、CG/音效/UI 提示。
- Action 必须返回可判断结果：成功、失败、无效、条件不足。
- 道具或 UI 不应绕过 Action 直接改业务状态。

## 4. 交互系统规范

### SimpleInteractable

每个可交互对象必须包含：

- `interactableId`
- 显示名与描述
- 普通/高亮表现资源
- 关联 Item 或运行时 Item 实例
- 工具效果映射
- 初始启用状态
- 可选分支、惩罚、联动规则

实现要求：

- `interactableId` 必须稳定且全局可定位。
- 场景加载后交互物应注册到 InteractionManager。
- 被拾取、破坏、禁用的交互物必须进入存档状态。

### 点击检视流程

标准流程：

1. 玩家点击交互物。
2. 输入层通知 `InteractionManager.OnInteractableClicked(SimpleInteractable)`。
3. GameManager 切换到 `Inspection`。
4. InteractionManager 通知 UIManager 打开 Inspector。
5. UI 展示名称、描述、可用 Action。
6. 检视结束后恢复 `Exploration`。

约束：

- 点击输入不得直接打开具体 UI 实现。
- UI 不得直接执行物品业务逻辑。

### 工具拖拽流程

标准流程：

1. 工具拖拽开始：`InteractionManager.OnToolDragStarted(ToolItem, Vector2)`。
2. 拖拽中：`InteractionManager.OnToolDragUpdated(Vector2)`。
3. 释放时检测目标：`CanUseToolOn(SimpleInteractable, ToolItem)`。
4. 有效则执行：`ExecuteToolInteraction(ToolItem, SimpleInteractable)`。
5. ToolItem 消耗耐久。
6. BranchManager / FlagManager / AudioManager / UIManager 接收结果。
7. 结束拖拽并恢复探索状态。

约束：

- 拖拽开始、更新、结束、取消必须完整实现。
- 工具有效性提示由 InteractionManager 判定，UIManager 只显示结果。
- 工具效果必须可配置，不应硬编码在 UI 或输入脚本中。

## 5. 配置表规范

配置源包括 SO、SO 列表、JSON、Excel 导表。建议每类配置只保留一个运行时真相来源。

### 必要配置类型

| 配置 | 建议结构 | 用途 |
| --- | --- | --- |
| GlobalConfig | SO | 初始 SAN、证据目标、工具槽、初始装备、默认耐久等全局常量 |
| SceneConfig | SO 列表 | 每个场景的背景、交互物、入口、音乐、环境参数 |
| InteractableConfig | SO 列表 | 交互物基础信息、描述、工具映射、拾取、惩罚、状态 |
| ToolEffectConfig | SO 字典 | 工具类型、效果 ID、效果数据映射 |
| BranchConfig | 嵌套配置 | 伪人概率、A/B 阈值、分支资源、惩罚 |
| PenaltyConfigTable | SO | SAN 惩罚、延迟、视觉效果、音效 ID |
| EvidenceConfig | SO | 证据 ID、来源、分值、类型、收集目标 |
| CGConfig | SO | CG 双版本、播放条件、前后事件 |
| RecordingConfig | SO | 录音 ID、真假版本、音频、时长、是否证据 |
| InteractableLinkageConfig | SO | 交互物联动规则 |
| VisualEffectConfig | SO | 视觉效果类型、参数、持续时间、资源替换 |

### InteractableConfig 字段

- 基础：`interactableId`, `displayName`, `normalSprite`, `highlightedSprite`
- 文本：`description`, `anomalyDescription`
- 工具映射：`List<ToolEffectEntry>`
- 工具字段：`toolType`, `effectId`, `consumesDurability`, `durabilityCost`, `triggersBranch`
- 拾取：`associatedItemId`, `isCollectible`
- 惩罚：`penaltyId`, `trigger`, `delaySeconds`
- 分支：`anomalyChance`, `spriteOverride`, `actionOverride`
- 状态：`initialActive`, `List<StateTransition>`
- 联动：`linkageRuleId`

约束：

- 配置字段名应与设计表保持一致或有明确映射。
- 新增配置字段时必须同步导入、运行时读取和存档兼容。
- 概率、分支、惩罚、视觉效果不得散落在业务代码中硬编码。

## 6. 存档规范

### 存档内容

必须覆盖：

- 当前场景
- GameState
- SAN 当前值和上限
- 证据进度
- 当前分支/伪人状态
- Inventory 内容
- 装备工具槽位
- 每个工具耐久
- FlagManager 字典
- 已拾取交互物 ID
- 已破坏/禁用交互物 ID
- 当前 VN 章节/序列/节点状态（如正在 VN 流程中）

### 读取流程

1. 读取 JSON 存档。
2. 恢复 Manager 基础状态。
3. 加载或确认目标场景。
4. 根据稳定 ID 恢复交互物状态。
5. 恢复库存、工具、flag、证据、VN 状态。
6. 刷新 UI。

约束：

- 不允许依赖场景临时对象引用做存档主键。
- 存档恢复后，所有 UI 应从 Manager 状态重新刷新。

## 7. UIManager 接口规范

UIManager 对外 API 应覆盖：

- `ShowPanel(string panelId)`
- `HidePanel(string panelId)`
- `HideAllPanels()`
- `ShowInspector(Item, SimpleInteractable)`
- `HideInspector()`
- `UpdateInspectorContent(Item)`
- `ShowToolDragIndicator(Sprite, Vector2)`
- `UpdateToolDragIndicator(Vector2)`
- `HideToolDragIndicator()`
- `ShowToolValidity(bool)`
- `UpdateSanDisplay(int current, int max)`
- `UpdateEvidenceDisplay(int current, int goal)`
- `UpdateContainmentDisplay(int current, int max)`
- `UpdateEquipmentSlots(...)`
- `PlaySanDamageEffect(int amount)`
- `PlayGlitchEffect(float duration)`
- `UpdateVignetteBySan(int current, int max)`
- `PlayScreenShake(float intensity, float duration)`
- `ShowHint(string text, float duration)`
- `ShowToolHint(string tool, string target, bool valid)`
- `ShowToolResult(string result, bool highlight)`
- `PlaySceneTransition(string type)`

约束：

- UIManager 方法应幂等，可重复调用。
- UIManager 只依赖 Core/Gameplay/Narrative 的抽象或数据类型，不依赖 Sample UI 类。
- 具体场景 UI 通过接口注册给 UIManager 调用。

## 8. 关卡运行流程规范

### 初始化阶段

标准顺序：

1. SceneManager 请求初始化。
2. GameManager 切换 `Init`。
3. GameLoopManager 启动目标关卡。
4. 注册场景交互物到 InteractionManager。
5. InventoryManager 初始化并装备初始工具。
6. SanityManager 初始化。
7. EvidenceManager 初始化。
8. FlagManager 设置初始关卡 flag。
9. UIManager 播放场景转场并刷新 HUD。
10. GameManager 切换 `Exploration`。
11. AudioManager 播放环境音。

### 交互阶段

- 玩家点击交互物时进入 Inspection。
- 玩家拖拽工具时进入 ToolDragging。
- 工具成功或失败后必须回调 InteractionManager，统一恢复状态。
- 分支判定、flag 写入、音效、UI 提示必须在同一次 Action 流程中完成。

约束：

- 初始化顺序必须稳定，避免 Manager 未注册时被调用。
- 每个关键阶段建议输出日志，便于复现问题。

## 9. 可复用视觉小说系统规范

### 分层

VN 系统分为四层：

1. 剧本数据层：ChapterConfig / JSON / Asset / Excel。
2. 剧本播放层：VNDirector / VNSequence / VNNode / VNBranch。
3. 桥接适配层：VNBridge / VNChoiceBridge / VNEventBridge。
4. 表现层：UIManager / CGMgr / AudioManager / FlagManager。

约束：

- 剧本数据层只描述“发生什么”，不描述具体 Manager 调用细节。
- 桥接层负责把脚本指令翻译为 Manager API。
- 表现层只执行显示、音频、CG、flag 等具体表现。

### 数据结构

- Chapter：章节。
- Sequence：段落/序列。
- Node：节点。
- Block：可选嵌套块。

Node 应支持：

- 说话人。
- 文本内容。
- 打字速度。
- 背景图。
- CG。
- 屏幕效果。
- 角色立绘与位置。
- 语音。
- 音效。
- BGM 切换。
- flag 条件。
- 等待玩家输入。
- 分支跳转。
- 默认下一节点。

Sequence 应支持：

- sequence ID。
- 播放前条件。
- 节点列表。
- 默认结束跳转。
- 结束条件或 flag 判断。

Chapter 应支持：

- chapter ID。
- 标题。
- 默认入口 sequence。
- sequence 列表。
- 章节开始前条件。
- 章节结束后行为。

### VNDirector 职责

- 维护当前章节、序列、节点。
- 维护等待输入状态。
- 维护自动播放状态。
- 读取 flag 并判断节点是否可播放。
- 执行节点指令。
- 处理顺序播放、跳转节点、跳转序列、跳转章节、等待选择。

约束：

- VNDirector 不直接依赖具体 UI 实现。
- VN 选择项必须通过 UIManager 回调到 VNDirector。
- 节点执行顺序必须可控；涉及音频、特效、等待输入的节点不得隐式跳过。

## 10. 编码规范

- 新增功能前先确认归属层级：Core、Gameplay、Narrative、Sample、Editor。
- Core 不得引用 Sample 命名空间。
- Gameplay 不得引用 Sample 命名空间。
- Sample 可以引用 Core/Gameplay/Narrative。
- Manager 公共 API 命名应表达业务意图，而不是 UI 或输入细节。
- 所有可配置 ID 使用字符串时必须集中定义或由配置生成，避免散落硬编码。
- 所有可存档对象必须有稳定 ID。
- 所有异步流程必须明确完成、取消和失败路径。
- UI 方法不承担业务判断，只显示传入状态。
- 调试 UI 不得成为正式流程依赖。
